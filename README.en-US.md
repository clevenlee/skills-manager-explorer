# Skills Manager Explorer

> **This project is a browser extension of upstream [https://github.com/xingkongliang/skills-manager](https://github.com/xingkongliang/skills-manager)**, providing a Web-side browse, compare, and scenario-assignment UI for a local Skills Manager SQLite database.
>
> 中文版：[README.md](./README.md)

A Bun-powered local fullstack web application that turns your local Skills Manager data (Skill, source, scenario) into something easier to read, and lets you safely adjust the scenario assignment of a single Skill without touching SQLite directly.

## Requirements

- macOS (the 1.0 verification platform)
- Bun 1.3.14 or a compatible higher version
- An existing Skills Manager SQLite database (generated by the upstream project)

This project does **not** require Node.js, does **not** run migrations, and does **not** create any Skills Manager business tables.

## Features

- Overview of total Skill, normalized source, scenario, and orphan Skill counts, with drill-down to multi-scenario Skills.
- Search, sort, paginated browsing for sources, scenarios, and Skills. Filter state is reflected in the URL so refresh, back, and detail-return are preserved.
- Comparison of any two sources, scenarios, or source-vs-scenario sets, showing intersection, left-only, right-only, and symmetric difference.
- Full-field Skill detail, with long text, hashes, and paths copy-to-clipboard.
- Single-skill scenario reassignment with double confirmation and optimistic conflict check; rolls back entirely on save failure.
- Language switcher in the top bar: defaults to Chinese, supports English; preference persists in the browser.

Scenarios and Skills data are **read-only**; the only write target is the `scenario_skills` join table.

## Local startup

```bash
bun install --frozen-lockfile
cp .env.sample .env
```

Edit `.env`, replace `SKILLS_MANAGER_DB` with the absolute path to your Skills Manager database, then:

```bash
bun run dev
```

- Web UI: http://127.0.0.1:5173
- Local API: http://127.0.0.1:4173/api/v1/status

The service binds only to loopback. `.env` and the database file are ignored by Git.

## How to use

1. Open **Overview** to see Skill / source / scenario / orphan Skill counts; click a card to drill in.
2. Open **Sources** or **Scenarios** to browse their lists and detail; click a name to enter a filtered Skill list.
3. In **Skills**, search by keyword, pick sources and scenarios, switch between grid/table view; open **Skill detail** to see all fields and edit scenario assignment.
4. In **Compare**, pick two sets (source or scenario) to see intersection, left-only, right-only, and symmetric difference; one-click swap.
5. Switch language via the bottom-bar language switcher; preference persists in the browser.

## Verification & contract

```bash
bun run verify          # format:check + lint + typecheck + test + openapi:check + openapi:lint + build
bun run test:coverage   # coverage
bun run test:e2e        # Playwright e2e
```

OpenAPI is generated from shared Zod schemas and the YAML is generated — never hand-edit:

```bash
bun run openapi:generate
bun run openapi:lint
bun run openapi:check
bun run mock
```

Prism listens on http://127.0.0.1:4010 by default, for contract-driven UI development.

## Production & single-file packaging

```bash
bun run build           # build web + embed assets + server
bun run start           # start the production server (web and /api/v1 share origin)
bun run package         # build a single-file binary for the current macOS platform
```

The single-file artifact lands at `dist/skills-manager-explorer`. The database path is still read from the runtime working-directory `.env` and is **not** bundled into the binary.

## Internationalization

- Default language: Chinese (`zh-CN`).
- Alternate language: English (`en-US`).
- Switching: take effect immediately via the top bar; preference persists in `localStorage.skillsManagerExplorer.locale`.
- Startup fallback order: localStorage → `navigator.languages` → `zh-CN`.
- Strings live in `src/web/i18n/locales/{zh-CN,en-US}.ts`. When adding new keys, **both** locales must be updated in the same change (key integrity enforced by `tests/unit/i18n-keys.test.ts`).
- See the 1.0.2 plan and todo for full i18n & naming conventions.
