# 技能管家浏览器 PRD 1.0

## 1. 文档信息

| 项目     | 内容                                                     |
| -------- | -------------------------------------------------------- |
| 产品名称 | 技能管家浏览器（skills-manager-explorer）                |
| 文档版本 | 1.0                                                      |
| 文档状态 | PRD 1.0 已实施并通过验收                                 |
| 产品形态 | 网页界面与本地服务                                       |
| 数据来源 | 用户通过 `.env` 配置的 Skills Manager 本地 SQLite 数据库 |

## 2. 背景与问题

Skills Manager 已将 Skill、来源、场景及其关联关系保存在本地 SQLite 数据库中，但现有 GUI 不便于完成以下工作：

- 快速查看当前安装了哪些 Skill，以及它们来自哪里。
- 判断各场景覆盖了哪些 Skill，发现没有归属任何场景的孤立 Skill。
- 比较两个来源、两个场景，或一个来源与一个场景之间的 Skill 差异。
- 在不直接操作 SQLite 的前提下，安全调整 Skill 的场景归属。

本产品面向个人使用，也面向团队成员使用。每位用户运行自己的本地服务，通过 `.env` 指向自己的数据库；产品不集中收集或同步团队数据库。

## 3. 问题陈述

如何让团队成员通过各自的本地数据库，快速浏览 Skill、判断场景覆盖、比较来源与场景差异，并安全调整 Skill 归属，而不必直接查询或修改 SQLite？

## 4. 目标用户与核心任务

### 4.1 目标用户

- 维护自己本地 Skills Manager 数据的开发者。
- 需要检查团队 Skill 来源和场景组织方式的技术负责人。

### 4.2 核心任务

当用户需要了解或整理本地 Skill 时，希望能够：

1. 快速找到目标 Skill，并查看其来源、描述、时间和所属场景。
2. 从来源或场景出发查看对应 Skill 集合。
3. 对两个 Skill 集合做交集和差集比较。
4. 找出未归属任何场景的 Skill。
5. 在确认后修改单个 Skill 的场景归属。

## 5. 产品目标与成功标准

### 5.1 产品目标

- 提供来源、场景和 Skill 的清晰浏览入口。
- 将来源筛选、场景筛选和集合比对从模糊的“高级搜索”变为可理解、可验收的操作。
- 将数据库写权限收窄到 `scenario_skills` 关联关系，降低误修改风险。
- 让不同团队成员只需修改 `.env`，即可使用同一套产品访问各自数据库。

### 5.2 成功标准

- 用户可从概览进入来源、场景、Skill 列表或比对工作台。
- 概览展示重复归属 Skill 数量，并可进入已启用重复归属筛选的 Skill 列表。
- 用户可通过关键词和多选筛选找到目标 Skill。
- 用户可查看任意两个来源或场景集合的共有项和独有项，也可跨类型比较来源与场景。
- 用户可一键查看全部孤立 Skill。
- 用户可修改并保存单个 Skill 的场景归属，保存失败时数据库不产生部分变更。
- 除 `scenario_skills` 外，产品不提供其他数据库写操作。
- 数据库配置错误、文件不可访问或表结构不兼容时，用户能看到可操作的诊断信息。

## 6. 产品原则

- **先看全局，再处理细节：** 使用“概览—浏览—比对—详情”的分层工作台，避免把所有能力堆入一个表格。
- **默认只读，写入收窄：** 浏览和比对均为只读；只有场景归属调整允许写入。
- **展示业务含义，不暴露数据库操作：** 用户面对的是来源、场景和 Skill，不需要理解 SQL 或关联表。
- **结果可解释：** 比对结果必须明确说明当前展示的是共有、仅左侧还是仅右侧 Skill。
- **本地优先：** 数据保留在用户设备上，不上传到远端服务。

## 7. 信息架构

### 7.1 一级入口

1. **概览**：展示关键数量和需要处理的异常项。
2. **来源**：浏览来源及各来源包含的 Skill。
3. **场景**：浏览场景及各场景包含的 Skill。
4. **Skill**：搜索、筛选、排序并进入详情。
5. **比对**：比较任意两个来源或场景所代表的 Skill 集合。

### 7.2 跳转原则

- 概览卡片可跳转至对应列表，并自动带入筛选条件。
- 点击来源或场景可进入其详情，也可跳转至已过滤的 Skill 列表。
- 来源、场景、Skill 列表中的数量和名称均应尽可能支持下钻。
- 比对结果中的 Skill 可进入 Skill 详情，返回时保留原比对条件和结果类型。

## 8. 功能需求

### 8.1 概览

概览至少展示：

- Skill 总数。
- 来源总数。
- 场景总数。
- 孤立 Skill 数，即没有任何 `scenario_skills` 关联记录的 Skill。

可补充展示以下状态，但不得影响 1.0 核心流程：

- 启用与停用 Skill 数量。
- 可更新、已是最新及检查异常的 Skill 数量。

交互要求：

- 点击 Skill、来源或场景数量进入对应列表。
- 点击孤立 Skill 数量进入 Skill 列表，并自动启用“未归属场景”筛选。
- 数据为空时展示空状态，不将其误报为数据库连接失败。

### 8.2 来源清单

来源数据由 `skills` 表聚合产生，不新增独立来源表。

列表至少展示：

- 来源名称，例如 `obra/superpowers`。
- 来源地址，例如 `https://github.com/obra/superpowers.git`。
- Skill 数量。
- 被场景引用的 Skill 数量。
- 孤立 Skill 数量。

交互要求：

- 支持按来源名称搜索。
- 支持按来源名称、Skill 数量排序。
- 点击来源进入来源详情，查看该来源的 Skill。
- 来源地址合法时可在新窗口打开；无法识别为安全 URL 时仅按文本展示。

### 8.3 场景清单

场景数据读取自 `scenarios` 表。

列表至少展示：

- 场景名称。
- 场景描述。
- 图标。
- Skill 数量。
- 排序值。

交互要求：

- 按 `sort_order` 升序展示；相同排序值按场景名称升序展示。
- 支持按场景名称搜索。
- 点击场景进入场景详情，查看其包含的 Skill。
- 1.0 不允许新增、编辑或删除场景。

### 8.4 Skill 清单

列表至少展示：

- 名称。
- 来源。
- 描述摘要。
- 创建时间。
- 更新时间。
- 所属场景；多个场景可分行或以标签展示。

展示规则：

- 描述在列表中截断展示，完整内容在详情中查看。
- 时间字段按毫秒级 Unix 时间戳解析，并按用户本地时区展示。
- 无场景的 Skill 显示明确的“未归属场景”状态，不留空。
- 字段为空或格式异常时显示占位状态，不能导致整页加载失败。
- 支持块状与表格两种展示方式；块状模式在宽屏默认一行展示 4 个 Skill，内容区空间足够时自动扩展为 5 个。

搜索、筛选与排序：

- 支持按 Skill 名称和描述进行关键词搜索。
- 支持来源多选筛选；同一维度内采用“满足任一所选来源”的语义。
- 支持场景多选筛选；默认采用“满足任一所选场景”的语义。
- 场景筛选提供“未归属场景”选项。
- 提供“仅重复归属”筛选，返回归属于两个及以上场景的 Skill；与“仅未归属”互斥。
- 来源筛选与场景筛选同时存在时，采用“来源条件并且场景条件”的语义。
- 支持按名称、来源、创建时间和更新时间排序，并明确升序或降序状态。
- Skill 列表提供创建时间与更新时间字段；创建时间映射数据库 `created_at`。
- 提供每页 20 条、每页 50 条与不分页选项，默认不分页。
- 更换搜索、筛选或排序条件后回到结果第一页。
- 无匹配结果时保留当前条件，并提供清空条件入口。

### 8.5 Skill 详情

详情展示 `skills` 表中的全部字段：

- `id`
- `name`
- `description`
- `source_type`
- `source_ref`
- `source_ref_resolved`
- `source_subpath`
- `source_branch`
- `source_revision`
- `remote_revision`
- `central_path`
- `content_hash`
- `enabled`
- `created_at`
- `updated_at`
- `status`
- `update_status`
- `last_checked_at`
- `last_check_error`

此外展示当前所属场景，并提供“调整场景归属”入口。

敏感路径类字段只用于本地页面展示，不发送到外部服务。长文本、哈希和路径应完整可复制。

### 8.6 调整 Skill 场景归属

1.0 只支持修改单个 Skill 的场景归属。

交互流程：

1. 用户从 Skill 详情进入编辑状态。
2. 页面展示全部已有场景，并勾选该 Skill 当前所属场景。
3. 用户增加或取消选择场景。
4. 用户点击保存后，页面明确展示待新增和待移除的关联，并请求确认。
5. 确认后，本地服务在单个数据库事务中更新 `scenario_skills`。
6. 保存成功后刷新详情、列表和概览中的相关数据。

约束与异常处理：

- 不允许通过该功能新增、编辑或删除场景。
- 不允许修改 `skills` 表中的任何字段。
- 后端必须校验 Skill 和场景均真实存在。
- 没有实际变化时禁用保存或直接提示无需保存。
- 保存失败时必须整体回滚，不得留下部分新增或部分删除的关联。
- 数据库被占用、只读或关联已变化时，应提示用户刷新后重试。
- 用户取消编辑时不得写入数据库。

### 8.7 来源与场景比对

比对工作台由左侧集合和右侧集合组成。每侧均可选择：

- 一个来源；或
- 一个场景。

因此支持：

- 来源与来源比较。
- 场景与场景比较。
- 来源与场景交叉比较。

比对结果提供四种视图：

| 视图     | 集合语义    | 用途                         |
| -------- | ----------- | ---------------------------- |
| 共有     | 左侧 ∩ 右侧 | 两侧都包含的 Skill           |
| 仅左侧   | 左侧 − 右侧 | 左侧包含、右侧不包含的 Skill |
| 仅右侧   | 右侧 − 左侧 | 右侧包含、左侧不包含的 Skill |
| 全部差异 | 左右对称差  | 只属于其中一侧的 Skill       |

交互要求：

- 选择条件后展示左右集合总数及各结果视图数量。
- 默认展示“共有”结果。
- 切换结果视图不清空左右选择。
- 支持交换左右集合。
- 比对结果使用统一 Skill 列表展示，并可继续按名称搜索和排序。
- 左右选择相同时允许比较，并正确显示共有项和空差异，不作为错误处理。
- 任一集合为空时仍展示可解释结果和数量。
- 1.0 不提供自由编写 SQL、布尔表达式或无限层级条件组的能力。

## 9. 数据规则

### 9.1 使用的表

| 表                | 用途              | 权限               |
| ----------------- | ----------------- | ------------------ |
| `skills`          | Skill 列表和详情  | 只读               |
| `scenarios`       | 场景列表和详情    | 只读               |
| `scenario_skills` | 场景与 Skill 关联 | 读；仅归属调整时写 |

### 9.2 来源标识与展示名称

来源分组必须使用稳定的规范化标识，不能默认通过“从 `source_ref` 末尾删除 Skill 名称”完成分组。

规则优先级：

1. `source_ref_resolved` 非空时，以其规范化结果作为来源标识。
2. 对 HTTP 或 HTTPS Git 地址，移除查询参数、片段、末尾 `/` 和末尾 `.git` 后参与分组。
3. 能识别为 GitHub 仓库地址时，展示名称取路径中的 `owner/repository`。
4. 无法识别为 GitHub 地址时，展示规范化地址或可读路径，不擅自截断目录。
5. 只有 `source_ref_resolved` 缺失时，才回退到 `source_ref`；若 `source_ref` 最后一个路径段与当前 Skill 的 `name` 完全相同，可移除该段作为候选来源，否则保留原值。
6. 无法得到来源时，统一归入“未知来源”，并保留原始字段供详情查看。

规范化只影响分组和展示，不修改数据库原始值。

### 9.3 场景关联

- 一个 Skill 可以属于零个或多个场景。
- “孤立 Skill”定义为在 `scenario_skills` 中不存在对应 `skill_id` 记录的 Skill。
- 同一 `scenario_id` 和 `skill_id` 组合不得重复。
- 调整归属时保留数据库对新增关联所需的默认值；若写入 `added_at`，应使用与现有数据库一致的毫秒级时间戳。

## 10. 本地服务与配置要求

### 10.1 配置

- 使用环境变量 `SKILLS_MANAGER_DB` 指定 SQLite 数据库绝对路径。
- 每位用户维护自己的本地 `.env`，产品不在网页中保存或分发团队成员的数据库路径。
- 环境变量缺失、路径不存在、文件不可读或必需表缺失时，本地服务不得进入正常运行状态。

### 10.2 启动校验

本地服务启动时至少校验：

- 数据库文件存在且可读。
- `skills`、`scenarios`、`scenario_skills` 三张表存在。
- 读取操作可执行。
- 需要调整归属时，数据库文件可写；不可写不影响浏览，但页面必须将编辑入口置为不可用并解释原因。

### 10.3 安全边界

- 本地服务默认只监听本机回环地址，不对局域网或公网开放。
- 所有 SQL 使用参数化查询。
- 前端不得接收任意 SQL 并交由后端执行。
- 除场景归属事务外，数据库连接不得执行其他新增、修改或删除操作。
- 本地数据库内容和路径不得发送至第三方分析、日志或外部 API。

## 11. 页面状态与错误处理

所有一级页面均需覆盖：

- 加载中。
- 正常有数据。
- 正常无数据。
- 搜索或筛选无结果。
- 数据库连接失败。
- 数据库结构不兼容。
- 数据读取失败。

写操作额外覆盖：

- 保存中，防止重复提交。
- 保存成功。
- 数据库只读。
- 数据库锁定或占用。
- 目标 Skill 或场景已不存在。
- 事务回滚后的失败提示。

错误信息应说明发生了什么、是否写入成功以及用户下一步可以做什么；不得只显示原始异常堆栈。

## 12. MVP 范围

### 12.1 本期实现

- `.env` 数据库路径配置与启动校验。
- 概览。
- 来源清单和来源详情。
- 场景清单和场景详情。
- Skill 清单、搜索、多选筛选和排序。
- Skill 全字段详情。
- 孤立 Skill 查询。
- 来源与场景的双集合比对。
- 单个 Skill 的场景归属调整。
- 基础加载、空状态和错误状态。

### 12.2 本期不做

- 新增、编辑或删除场景：本期只管理既有场景与 Skill 的关联。
- 新增、编辑、删除、安装或更新 Skill：避免把浏览器扩展成完整 Skills Manager。
- 批量修改 Skill 场景归属：先验证单条调整的安全性和使用频率。
- 团队数据库汇总、账号体系、云同步和远程访问：团队成员使用各自的本地数据库。
- 自定义 SQL、任意布尔表达式和无限条件组：双集合比对已覆盖 1.0 的核心差异分析任务。
- 修改来源数据：来源由 `skills` 表聚合，只读展示。
- 第三方遥测和外部数据上报：保持本地优先。

## 13. 关键假设与验证方式

### 13.1 必须成立

- 团队成员的 Skills Manager 数据库至少包含三张必需表及本文依赖字段。进入开发前应收集不同成员的只读表结构并做兼容性比对。
- 用户运行产品时允许启动本地服务。通过最小启动原型验证开发和使用流程。
- 调整场景归属是高价值且可接受的唯一写操作。使用可回滚的数据库副本验证增删关联和失败回滚。

### 13.2 应当成立

- 用户能理解共有、仅左侧、仅右侧和全部差异。通过页面文案和结果数量降低集合术语门槛。
- `source_ref_resolved` 在大多数数据中可稳定代表仓库来源。使用团队样本验证规范化后是否发生误合并或重复来源。
- 用户更常调整单个 Skill，而不是批量调整。上线后根据实际反馈决定是否扩展批量操作。

### 13.3 暂不验证

- 用户是否需要跨设备同步比对条件。
- 用户是否需要保存常用查询或导出结果。
- 用户是否需要写入审计记录。

这些能力不影响 1.0 核心价值，暂不扩大范围。

## 14. 风险与应对

| 风险                                    | 影响                     | 应对                                                 |
| --------------------------------------- | ------------------------ | ---------------------------------------------------- |
| Skills Manager 升级后数据库结构变化     | 页面或写操作失效         | 启动时检查表与字段，错误时明确报告不兼容项           |
| 来源地址格式不一致                      | 来源被重复分组或错误合并 | 使用稳定规范化规则，并保留原始字段供核对             |
| Skills Manager GUI 与本产品同时写数据库 | 保存冲突或数据库锁定     | 使用短事务；失败时回滚并提示刷新后重试               |
| 写权限范围失控                          | 用户本地数据被意外修改   | 后端按操作白名单限制，只允许更新关联表               |
| 高级筛选语义不清                        | 相同条件得到非预期结果   | 明确同维度为“或”、跨维度为“且”，比对使用固定集合语义 |
| 功能全部堆在 Skill 表格                 | 页面复杂、难以学习       | 采用概览、浏览、比对、详情分层入口                   |

## 15. 验收标准

### 15.1 配置与连接

- [x] 配置有效数据库路径后，本地服务可读取三张必需表并进入应用。
- [x] 未配置路径、路径无效或必需表缺失时，页面展示具体原因和修复建议。
- [x] 数据库只读时，所有浏览与比对功能可用，归属编辑入口不可用并说明原因。

### 15.2 浏览

- [x] 概览中的 Skill、来源、场景和孤立 Skill 数量与数据库查询结果一致。
- [x] 概览中的重复归属 Skill 数量与归属于两个及以上场景的 Skill 数量一致，并可下钻查看。
- [x] 来源清单按照规范化来源正确聚合，并可进入对应 Skill 列表。
- [x] 场景清单及场景详情与 `scenarios`、`scenario_skills` 数据一致。
- [x] Skill 列表正确展示多个场景和未归属状态。
- [x] Skill 列表支持块状与表格展示、重复归属筛选，以及默认不分页的 20/50 条分页选择。
- [x] 关键词、来源多选、场景多选和未归属筛选可组合使用，组合语义符合本文定义。
- [x] 名称、来源、创建时间和更新时间排序结果正确。
- [x] Skill 列表展示创建时间，并支持创建时间、更新时间等排序字段的升序和倒序。
- [x] Skill 详情展示 `skills` 表全部字段，空值和长文本可正常处理。

### 15.3 比对

- [x] 支持来源—来源、场景—场景和来源—场景三种比较组合。
- [x] 共有、仅左侧、仅右侧和全部差异的数量及明细与集合计算结果一致。
- [x] 左右相同、集合为空和无差异场景均有正确结果，不被当作异常。
- [x] 交换左右集合后，仅左侧和仅右侧结果同步互换。

### 15.4 场景归属调整

- [x] 用户可以为单个 Skill 增加或移除已有场景。
- [x] 保存前展示明确的变更摘要并再次确认。
- [x] 保存成功后，详情、列表、场景数量和孤立 Skill 数量保持一致。
- [x] 保存失败时事务整体回滚，不出现部分变更。
- [x] 归属调整不会修改 `skills`、`scenarios` 或其他表的数据。

## 16. 后续决策点

以下事项不阻塞 1.0 进入执行计划，但应在开发验证或首轮试用后决定：

- 是否增加批量调整场景归属。
- 是否保存常用筛选或比对条件。
- 是否支持导出筛选和比对结果。
- 是否利用现有 `audit_log` 记录归属调整历史。
- 团队实际最大数据规模及对应性能基线。
