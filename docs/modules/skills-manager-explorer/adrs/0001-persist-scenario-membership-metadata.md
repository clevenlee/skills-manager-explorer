# ADR-0001：同步 Skills Manager 场景归属元数据

## 状态

已接受。

## 日期

2026-07-18。

## 背景

Skills Manager 1.28.3 的 SQLite 数据库不是场景归属的唯一持久化来源。主程序会把 `skills/.skills-manager/scenario-skills/` 下的关联文件用于备份与启动重建，并且运行中的 GUI 不会自动刷新由外部进程修改的内存缓存。

本项目原先只更新 `scenario_skills`。SQLite 事务虽然已经提交，但对应元数据未更新时，GUI 运行期间看不到变更；主程序重启后还可能使用旧元数据覆盖数据库关联。

进一步验证发现，关联元数据里的空 `tools` 对象并不等于“没有工作区”。Skills Manager GUI 启动时会为当前场景中缺失的工具开关补入默认启用值，随后把 Skill 同步到对应工具目录。因此，只写关联和空工具映射仍可能产生延迟的工作区副作用。

## 决策

- `scenario_skills` 保存 Skill–场景归属。
- 对每个新增归属，在 `scenario_skill_tools` 中为当前数据库已知的工具写入显式禁用值；这不是应用工作区，而是阻止 GUI 把缺失值解释为默认启用。
- 当数据库旁存在完整 Skills Manager 元数据快照时，批量添加归属必须同步创建对应关联文件。
- 关联文件的 `tools` 必须保存同一组显式 `false`，保证 GUI 重建数据库后保护值不丢失；不得使用会触发默认启用的空对象。
- 所有数据库新增关联置于同一个即时事务中。
- 元数据文件通过同目录临时文件和原子硬链接创建，不覆盖既有文件。
- 文件写入失败或 SQLite 提交失败时，删除本次已创建的元数据文件并回滚数据库事务。
- 不得修改 `active_scenario`、`settings`、`skill_targets`、`projects`、`audit_log`、Skill/场景实体，也不得调用同步或应用场景逻辑、创建或删除任何工作区文件。

## 备选方案

### 仅写 SQLite

实现简单，但无法跨 Skills Manager GUI 重启可靠保留，因此拒绝。

### 调用 Skills Manager CLI

CLI 能使用主程序共享核心写入元数据，但无法从任意 `SKILLS_MANAGER_DB` 路径可靠推导其运行仓库，而且批量跨场景调用不具备单事务语义，因此暂不采用。

### 保存空 `tools` 对象

结构上与主程序新增归属时的元数据相似，但 GUI 启动会把缺失开关补成启用，并可能同步当前场景，因此拒绝。

## 影响

- 使用 Skills Manager 1.28.3 完整元数据快照的用户，批量归属可在 GUI 刷新或重启后保留。
- 新增归属默认不绑定任何已知工作区；用户之后必须通过 Skills Manager 的显式工作区/应用操作完成部署。
- 临时测试数据库和不含元数据快照的旧版本数据库仍只写 SQLite。
- 自动化测试必须覆盖元数据成功落盘、工作区保护值、非授权表不变及元数据失败时的数据库回滚。
